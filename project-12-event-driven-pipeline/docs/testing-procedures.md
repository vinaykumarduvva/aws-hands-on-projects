# Testing Procedures: Event-Driven Pipeline

After deploying the pipeline via the console or CLI, run the following test suites to validate functionality, security, and resiliency.

---

## 🧪 1. Functional Tests (End-to-End Pipeline)

This test validates the "Happy Path": S3 → SQS → Lambda → S3.

### Step 1: Create a Test File
Create a simple `test-data.csv` locally:
```csv
id,name,role
1,Alice,Admin
2,Bob,User
3,Charlie,User
```

### Step 2: Upload to S3
Navigate to the S3 Console, open your `event-pipeline-source-[ACCOUNT_ID]` bucket, create a folder named `uploads/`, and upload the `test-data.csv` file into it. 

*CLI Alternative:*
```bash
aws s3 cp test-data.csv s3://event-pipeline-source-[YOUR-ACCOUNT-ID]/uploads/test-data.csv
```

### Step 3: Verify Output
1. Navigate to the `event-pipeline-output-[ACCOUNT_ID]` bucket.
2. Verify that a new file exists: `processed/test-data.csv-result.json`.
3. Download and open the file. You should see a JSON summary generated by Lambda:
```json
{
  "source_bucket": "event-pipeline-source-[ACCOUNT_ID]",
  "source_key": "uploads/test-data.csv",
  "processed_at": "2026-07-16T12:00:00Z",
  "status": "SUCCESS",
  "row_count": 4
}
```

## 🛑 2. Dead Letter Queue (DLQ) / Resiliency Test

This test simulates a "Poison Pill" — a file that causes the Lambda function to crash continuously. We want to ensure that after 3 failed retries, SQS automatically moves the message to the Dead Letter Queue.

### Step 1: Force a Lambda Failure
Modify the Lambda function code to intentionally crash. Go to the Lambda Console, find the `file-processor` function, and add this line right below `def lambda_handler(event, context):`:

```python
def lambda_handler(event, context):
    raise Exception("Simulated fatal error for DLQ testing!")
    # ... rest of code
```
Click **Deploy**.

### Step 2: Trigger the Pipeline
Upload a new file `crash-test.csv` to the source bucket's `uploads/` folder.

### Step 3: Monitor the Queues
1. Navigate to the **SQS Console**.
2. Select `file-processing-queue`. You will see the **Messages available** count spike to `1`, then drop to `0` (as Lambda grabs it), and then return to `1` (because Lambda crashed and the message went back to the queue).
3. Wait approximately 1-2 minutes for SQS to retry the delivery 3 times.
4. Refresh the console. You should now see **Messages available** on the `file-processing-queue` drop to `0`, and **Messages available** on `file-processing-dlq` rise to `1`.

*Conclusion:* The DLQ is successfully capturing failed messages, preventing the pipeline from being perpetually blocked!

> [!TIP]
> Remember to remove the `raise Exception` line and redeploy your Lambda function after this test.

## 🔒 3. Security Tests (Negative Testing)

### Test A: Unauthorized SQS Publish
Attempt to send a message to the SQS queue directly from an unauthorized source (like your personal IAM user, bypassing S3).

```bash
aws sqs send-message \
    --queue-url https://sqs.ap-south-1.amazonaws.com/[ACCOUNT_ID]/file-processing-queue \
    --message-body "Hack attempt"
```
**Expected Result:** `AccessDenied` error, proving the Resource Policy correctly restricts publishing to the specific S3 bucket.

### Test B: Public Access Verification
Attempt to read the uploaded test file anonymously via the public internet URL.
1. In the S3 Console, click on `uploads/test-data.csv`.
2. Copy the **Object URL** and paste it into an incognito browser window.
**Expected Result:** An XML `AccessDenied` response, verifying Block Public Access is active.
